# OCR 비자 신청 시스템 프론트 기획서

작성일: 2026-05-08

## 1. 목적

이 문서는 1차 내부 운영용 OCR 비자 신청 시스템의 프론트 요구사항을 화면 기준으로 다시 정리한 문서다.  
현재 단계에서는 실제 API 연동보다 `화면 흐름`, `사용자별 동선`, `더미 데이터 구조`, `향후 필요한 도메인/테이블 후보`를 명확하게 잡는 것이 목적이다.

## 2. 현재 범위

- 프론트는 더미 데이터 기반으로 우선 구현한다.
- 백엔드 API 연동은 지금 단계에서 하지 않는다.
- OCR 결과를 실제로 호출하지 않고, 결과가 들어온 뒤 어떤 화면으로 보여줄지를 먼저 설계한다.
- 사용자 타입은 `학생`, `학교`, `유학원` 3종이다.

## 3. 공통 업무 규칙

### 신청 상태

현재 화면에서는 아래 2개 상태만 우선 사용한다.

- `보완`
- `완료`

향후 전체 상태 확장 후보는 다음과 같다.

- `신청`
- `접수`
- `보완`
- `완료`
- `반려`

### 신청 구분

- 신청은 `신규 신청` 또는 `연장 신청`으로 구분된다.
- 실제 서류 요구사항은 신청명보다 `비자 유형`에 따라 달라진다.

## 5. 사용자별 화면 구조

### 5-1. 로그인 페이지

공통 요구:

- 로그인 타입 선택이 먼저 필요하다.
- 타입에 따라 입력 폼과 진입 화면이 달라진다.

학생 로그인:

- 국적 드롭다운
- 여권번호
- 생년월일

학교 로그인:

- 아이디
- 비밀번호

유학원 로그인:

- 아이디
- 비밀번호

### 5-2. 학생 화면

#### 학생 메인 페이지

- 신청 건 리스트를 카드 또는 리스트 형태로 노출
- 각 신청건에 포함할 정보
  - 신청 유형
  - 비자 타입
  - 신청 날짜
  - 신청 상태
- `파일 추가` 버튼은 우선 UI만 제공하고 기능은 연결하지 않음

#### 학생 신청 상세 페이지

- 선택한 신청건 기준으로 상세 정보 진입
- 해당 신청 유형/비자 타입에 필요한 서류 목록 표시
- 각 서류별 제출 상태 표시
  - 제출
  - 미제출

### 5-3. 학교 화면

학교는 우선 단일 페이지 구조로 간다.

#### 학생 목록 페이지

- 학생 목록 테이블 또는 리스트
- 검색 기능
- 상태 또는 비자 유형 기준 필터
- 상단 `학생 명단표 추출` 버튼 제공
  - 현재는 버튼 UI만 제공
  - 실제 CSV 다운로드 기능은 추후 연결

### 5-4. 유학원 화면

#### 유학원 메인 대시보드

- 상단 요약 카드
  - 전체 문서 개수
  - 완료 개수
  - 보완 개수
  - 진행 중 신청건 수
- 학생 검색 기능
- 신청건 기준 리스트
  - 같은 학생도 여러 신청건을 가질 수 있음
- 리스트에 포함할 정보
  - 학생 이름
  - 국적
  - 생년월일
  - 비자 신청 날짜
  - 신청 유형
  - 비자 타입
  - 신청 상태
- `단체수납표 추출` 버튼 제공
  - 현재는 버튼 UI만 제공

#### 유학원 신청 상세 페이지

- 좌측
  - 학생 기본 정보
  - 신청 기본 정보
  - 필요 문서 목록
  - 각 문서의 제출/미제출 상태
- 우측
  - 좌측 문서 목록에서 선택한 문서의 미리보기 영역
  - 실제 파일 뷰어 대신 더미 썸네일/텍스트 패널로 우선 구성 가능

#### ZIP 업로드 영역

- 별도 화면 또는 토글 패널로 제공 가능
- ZIP 업로드 버튼 필요
- ZIP 업로드 처리 규칙을 안내 문구로 명확히 보여줘야 함

## 6. ZIP 업로드와 OCR 처리 규칙

이 부분은 나중에 실제 OCR 파이프라인과 연결될 핵심 규칙이다. 프론트에서는 안내 문구와 처리 흐름을 보여주는 수준으로 먼저 반영한다.

### 학생 구간 분리 규칙

- 스캔본 ZIP 안의 문서들은 `통합신청서`를 기준으로 학생 단위를 나눈다.
- 첫 번째 통합신청서가 나오면 그 뒤 문서들은 같은 학생 자료로 본다.
- 다음 통합신청서가 나오면 그 직전까지를 이전 학생 자료로 마감한다.
- 두 번째 통합신청서부터는 새로운 학생 세트가 시작된다.

### 학생 정보 추출 우선순위

1. `여권`
   - 학생 이름
   - 국적
   - 생년월일
2. `통합신청서`
   - 여권에서 누락되거나 불확실한 정보 보완
   - 외국인등록번호
   - 전화번호
3. `부동산계약서`
   - 주소 추출

정리하면:

- 학생 구간 분리는 `통합신청서`
- 기본 인적사항은 `여권 우선`
- 부족한 값은 `통합신청서 보완`
- 주소는 `부동산계약서`

## 8. 향후 필요한 테이블/도메인 후보

프론트 요구사항 기준으로 역으로 보면, 나중에 백엔드에서 최소 아래 정도는 필요할 가능성이 높다.

### 핵심 엔티티 후보

- `user`
- `student`
- `school`
- `agency`
- `application_case`
- `visa_type`
- `document_requirement`
- `submitted_document`
- `upload_batch`
- `ocr_extraction_result`
- `review_task`

### 이유

- 같은 학생이 여러 신청건을 가질 수 있으므로 `student`와 `application_case`는 분리해야 한다.
- 비자 유형별 필요 서류가 다르므로 `visa_type`과 `document_requirement` 매핑이 필요하다.
- 제출 여부와 실제 업로드 파일은 별도 관리가 필요하므로 `submitted_document`가 필요하다.
- ZIP 업로드 단위와 학생 단위가 다르므로 `upload_batch`를 분리하는 편이 맞다.
- OCR 추출 원문과 정규화된 학생 정보는 나중에 분리 저장하는 편이 안전하다.
